# ВВЕДЕНИЕ

Целью настоящей курсовой работы является создание информационной системы, представленной в виде клиент-серверного веб-приложения, предназначенного для автоматизации процессов учета успеваемости и посещаемости студентов в высшем учебном заведении. В процессе выполнения курсовой работы были последовательно пройдены все этапы жизненного цикла программного обеспечения: разработка технического задания, анализ рисков, проектирование структурной, функциональной и информационной моделей, непосредственная разработка программного продукта, а также его тестирование и доработка.

Курсовая работа направлена на закрепление теоретических знаний, полученных в ходе изучения дисциплин, связанных с программированием, проектированием программных систем и разработкой пользовательского интерфейса, а также на развитие практических навыков работы с современными инструментами и технологиями веб-разработки, включая серверные фреймворки, клиентские библиотеки, системы управления базами данных и средства обеспечения информационной безопасности.

Разрабатываемая информационная система представляет собой электронный журнал успеваемости, функционирующий в архитектуре клиент-сервер и обеспечивающий взаимодействие между тремя основными группами пользователей: преподавателями, студентами и административным персоналом. Система поддерживает ведение двух типов журналов — журнала успеваемости и журнала посещаемости — по каждой дисциплине и группе, обеспечивает автоматический пересчет статистических показателей, предоставляет инструменты фильтрации и поиска, а также включает механизмы резервного копирования и восстановления базы данных.

В рамках проекта реализовано четыре роли пользователей с разграничением прав доступа: администратор, осуществляющий управление учетными записями, справочниками и назначениями преподавателей; преподаватель, ведущий журналы успеваемости и посещаемости по закрепленным за ним дисциплинам и группам; студент, просматривающий собственные оценки и статистику посещаемости; сотрудник деканата, формирующий сводные отчеты по группам и дисциплинам.

Основные функциональные возможности системы включают: авторизацию и аутентификацию пользователей с использованием JWT-токенов, хранящихся в защищенных httpOnly-cookie; создание и управление учебными группами, дисциплинами и назначениями преподавателей; автоматическое формирование парных журналов (успеваемости и посещаемости) при назначении преподавателя; выставление оценок с поддержкой различных систем оценивания и типов работ; учет посещаемости с возможностью отметки статусов «присутствует», «отсутствует», «уважительная причина»; автоматический расчет среднего балла, процента посещаемости и итоговых оценок; ведение журнала аудита для фиксации всех изменений данных; резервное копирование и восстановление базы данных с использованием скриптов PowerShell.

В рамках данной информационной системы не реализованы следующие возможности: интеграция с внешними учебными платформами (LMS Moodle, 1С:Университет), автоматическая рассылка уведомлений пользователям по электронной почте, мобильное приложение, экспорт отчетов в форматы PDF и Microsoft Excel, автоматическое формирование приказов о переводе и отчислении студентов.

# 1 ОЦЕНКА РИСКОВ

## 1.1 Оценка рисков для построения информационной системы

В процессе эксплуатации информационной системы учета успеваемости студентов, как и любой другой системы, обрабатывающей персональные данные и функционирующей в сетевой среде, существует ряд потенциальных угроз, реализация которых может привести к нарушению конфиденциальности, целостности или доступности информации. Для систематизации и приоритизации данных угроз был проведен анализ рисков с привлечением методов экспертной оценки.

На основании анализа предметной области, архитектурных особенностей приложения и специфики обрабатываемых данных были выделены следующие риски:

- Нарушение конфиденциальности персональных данных студентов и преподавателей, включая фамилии, имена, отчества, учетные данные и сведения об успеваемости, что может повлечь за собой ответственность в соответствии с ФЗ-152 «О персональных данных»;
- Отказ системы резервного копирования и восстановления базы данных, в результате чего при сбое оборудования или программного обеспечения может произойти необратимая утрата всех накопленных сведений об оценках и посещаемости;
- Хищение сессионных JWT-токенов и несанкционированный доступ к системе от имени легитимного пользователя, что позволит злоумышленнику просматривать или модифицировать чужие данные;
- Несанкционированный доступ к функциям администратора вследствие ошибок в контроле доступа на уровне объектов API, например, путем прямого обращения к защищенным конечным точкам без соответствующей роли;
- Уязвимость к атакам типа «Отказ в обслуживании» (DoS), при которой многократная отправка запросов на авторизацию или получение данных может привести к исчерпанию ресурсов сервера и недоступности системы для легитимных пользователей;
- Утечка технической информации об архитектуре системы, используемых технологиях и версиях программного обеспечения через сообщения об ошибках или открытые репозитории;
- Отказ системы при пиковых нагрузках, возникающих в периоды массового просмотра журналов перед сессией или одновременного ввода оценок несколькими преподавателями;
- Низкое качество кода и ошибки в бизнес-логике, в частности некорректный пересчет среднего балла, итоговой оценки или процента посещаемости, что может привести к искажению академических показателей студентов;
- Потеря данных при одновременном редактировании одного и того же журнала несколькими преподавателями вследствие отсутствия механизмов блокировки записей;
- Несанкционированное повышение привилегий путем манипуляции идентификаторами пользователей в запросах к API, если сервер не осуществляет достаточную проверку прав на уровне каждой операции.

Выделенные риски охватывают ключевые аспекты информационной безопасности — конфиденциальность, целостность и доступность данных, а также корректность функционирования бизнес-логики приложения. При их определении учитывались предполагаемая структура системы, функциональные требования, состав обрабатываемых данных и этапы разработки программного продукта.

# 2 ПРОЕКТИРОВАНИЕ

## 2.1 Описание процессов системы в методологии функционального моделирования IDEF0

Функциональное моделирование системы выполнено в методологии IDEF0, которая позволяет представить деятельность системы в виде иерархии взаимосвязанных функциональных блоков с четким определением входов, управлений, механизмов и выходов каждого процесса. Данная методология обеспечивает наглядное представление архитектуры системы и служит основой для последующего проектирования информационной модели и базы данных.

На рисунке 2.1 представлена контекстная диаграмма «черный ящик» в нотации IDEF0. Основной функцией моделируемого процесса является «Работа с электронным журналом». Процесс начинается с перехода пользователя по URL-адресу информационной системы, после чего система обеспечивает взаимодействие между преподавателями, студентами, администраторами и сотрудниками деканата в рамках учета успеваемости и посещаемости.

Для корректного функционирования информационной системы необходимо обеспечение определенных входных данных и управляющих воздействий. В качестве входных данных выступают сведения о студентах, преподавателях, дисциплинах, учебных группах, а также непосредственно оценки и отметки о посещаемости. Управляющие воздействия включают требования ФЗ-152 «О персональных данных», руководство пользователя, ролевую модель доступа, требования к доступности сервера и базы данных, а также необходимость наличия активной сессии пользователя. Механизмами выполнения функции являются администратор, преподаватель, студент и сотрудник деканата, которые осуществляют взаимодействие с системой через веб-интерфейс.

В результате выполнения основной функции формируются выходные данные: измененные записи в базе данных (оценки, посещаемость, назначения), статистика успеваемости по группе, сводные отчеты для деканата, а также резервные копии базы данных. После завершения работы пользователь осуществляет выход из системы, при этом все внесенные изменения сохраняются в базе данных.

Для детального представления работы информационной системы была выполнена декомпозиция контекстной диаграммы. Основная функция разбита на восемь взаимосвязанных подфункций, каждая из которых отвечает за конкретный этап взаимодействия пользователя с электронным журналом. Декомпозиция представлена в приложении Г.

Описание подфункций декомпозиции:

- **А1 — Авторизоваться в системе.** Пользователь вводит логин и пароль на странице авторизации. Система проверяет учетные данные в таблице users, формирует JWT-токен и устанавливает его в httpOnly-cookie. При успешной авторизации пользователь перенаправляется в личный кабинет в соответствии со своей ролью. Управляющими воздействиями являются политика безопасности, требования к сложности пароля и ограничение частоты попыток входа.
- **А2 — Управление учетными записями.** Администратор осуществляет создание, редактирование и удаление учетных записей пользователей всех ролей. Подфункции включают: А2.1 — создание пользователя с указанием логина, ФИО, пароля и роли; А2.2 — редактирование данных пользователя, включая изменение группы студента или кафедры преподавателя; А2.3 — удаление пользователя из системы. Выходом являются обновленные записи в таблице users и связанных таблицах students, teachers.
- **А3 — Управление справочниками и назначениями.** Администратор управляет учебными группами, дисциплинами и назначениями преподавателей. Подфункции включают: А3.1 — создание и редактирование групп; А3.2 — создание и редактирование дисциплин; А3.3 — назначение преподавателя на группу и дисциплину с указанием семестра. При создании назначения система автоматически формирует два журнала: журнал успеваемости (type='grades') и журнал посещаемости (type='attendance').
- **А4 — Ведение журнала успеваемости.** Преподаватель открывает закрепленный за ним журнал успеваемости, создает учебные работы (works) с указанием типа, системы оценивания, максимального балла и срока сдачи, а также занятия (lessons). Преподаватель выставляет оценки (grades) студентам за каждую работу. Система автоматически пересчитывает средний балл и итоговую оценку по дисциплине. Входными данными являются списки студентов группы, типы работ и системы оценивания; выходными — записи в таблицах works, lessons, grades.
- **А5 — Ведение журнала посещаемости.** Преподаватель открывает журнал посещаемости, создает занятия и отмечает статус посещаемости каждого студента: present (присутствует), absent (отсутствует), excused (уважительная причина). Система автоматически подсчитывает процент посещаемости для каждого студента по дисциплине.
- **А6 — Просмотр успеваемости и посещаемости.** Студент просматривает свои собственные оценки и статистику посещаемости по всем дисциплинам. Преподаватель и сотрудник деканата могут просматривать полные журналы с фильтрацией по группе, дисциплине, семестру и преподавателю.
- **А7 — Формирование отчетов и статистики.** Сотрудник деканата формирует сводные отчеты об успеваемости и посещаемости по группам и дисциплинам. Система агрегирует данные из таблиц grades и attendances, рассчитывает средние показатели и формирует статистику в табличном виде.
- **А8 — Резервное копирование и восстановление данных.** Администратор или автоматизированный процесс запускает PowerShell-скрипт backup.ps1, который создает SQL-дамп базы данных с временной меткой в имени файла. При необходимости восстановления администратор запускает скрипт restore.ps1, который интерактивно предлагает выбрать резервную копию, создает предварительную копию текущего состояния, удаляет и пересоздает базу данных, после чего восстанавливает данные из выбранного файла.

## 2.2 Информационная модель

Диаграмма потоков данных (DFD), представленная в приложении В, описывает основные процессы обработки информации в приложении, начиная с авторизации пользователя и заканчивая формированием статистических отчетов. Модель демонстрирует, как данные перемещаются между внешними сущностями, процессами обработки и хранилищами данных, что позволяет наглядно представить информационные потоки внутри системы и взаимосвязи между ее компонентами. [2]

*[ВСТАВИТЬ: Рисунок 2.2 — Диаграмма потоков данных (файл требования/dfd.png)]*

Основные внешние сущности, взаимодействующие с системой:

- **Администратор.** Пользователь с наивысшим уровнем привилегий, обладающий правами на управление учетными записями, справочниками, назначениями преподавателей, просмотр журнала аудита и управление режимом технического обслуживания.
- **Преподаватель.** Пользователь, осуществляющий ведение журналов успеваемости и посещаемости по закрепленным за ним дисциплинам и группам. Имеет право создавать работы и занятия, выставлять оценки, отмечать посещаемость.
- **Студент.** Пользователь, имеющий доступ только к собственным академическим данным: оценкам по дисциплинам, статистике посещаемости, списку своих журналов.
- **Сотрудник деканата.** Пользователь с правами просмотра всех журналов и формирования сводных отчетов по группам и дисциплинам без возможности редактирования оценок.

Основные процессы обработки данных:

- **Процесс 1 — Авторизация и аутентификация.** Неавторизованный пользователь передает логин и пароль. Система проверяет учетные данные в хранилище D1 (Пользователи), формирует JWT-токен и устанавливает его в httpOnly-cookie. При превышении лимита попыток входа IP-адрес временно блокируется.
- **Процесс 2 — Управление пользователями.** Администратор создает, редактирует или удаляет учетные записи. Данные записываются в хранилище D1, а изменения фиксируются в хранилище D14 (Журнал аудита).
- **Процесс 3 — Управление справочниками.** Администратор управляет группами (D3), дисциплинами (D4) и назначениями преподавателей (D5). При создании назначения система автоматически порождает записи в хранилище D6 (Журналы).
- **Процесс 4 — Ведение журнала успеваемости.** Преподаватель создает работы (D7) и занятия (D8) в рамках журнала успеваемости (D6), после чего выставляет оценки (D9). Система производит автоматический пересчет среднего балла и итоговой оценки.
- **Процесс 5 — Ведение журнала посещаемости.** Преподаватель создает занятия (D8) в журнале посещаемости (D6) и фиксирует статус посещаемости каждого студента (D10). Система рассчитывает процент посещаемости.
- **Процесс 6 — Просмотр данных.** Студент запрашивает собственные оценки из D9 и посещаемость из D10. Преподаватель и деканат просматривают полные журналы с применением фильтров.
- **Процесс 7 — Формирование отчетов.** Сотрудник деканата запрашивает агрегированные данные из D9 и D10. Система формирует сводную статистику по группам и дисциплинам.
- **Процесс 8 — Резервное копирование и восстановление.** Администратор инициирует создание резервной копии путем запуска PowerShell-скрипта backup.ps1. Скрипт автоматически обнаруживает утилиту pg_dump в системе (через Get-Command или перебор стандартных путей установки PostgreSQL), считывает параметры подключения из файла server/.env, устанавливает переменную окружения PGPASSWORD и формирует SQL-дамп в формате plain text (ключ -F p) с именем, содержащим временную метку. При необходимости восстановления администратор запускает скрипт restore.ps1, который интерактивно выводит нумерованный список доступных резервных копий из директории backups/, предлагает выбрать нужную, создает предварительную резервную копию текущего состояния базы данных (для возможности отката), принудительно удаляет существующую базу данных с помощью утилиты dropdb (ключ --force завершает активные соединения), пересоздает пустую базу через createdb и восстанавливает данные из выбранного файла с помощью утилиты psql.

Основные хранилища данных:

- **D1 — Пользователи.** Содержит учетные записи всех пользователей системы с хэшированными паролями, ФИО, ролями и статусом блокировки.
- **D2 — Роли.** Справочник ролей: admin, teacher, student, deanery.
- **D3 — Группы.** Справочник учебных групп с указанием года поступления.
- **D4 — Дисциплины.** Справочник учебных дисциплин.
- **D5 — Назначения преподавателей.** Связь преподавателя с группой, дисциплиной и семестром.
- **D6 — Журналы.** Журналы успеваемости и посещаемости, автоматически создаваемые при назначении.
- **D7 — Работы.** Учебные работы в рамках журнала: контрольные, лабораторные, экзамены с параметрами оценивания.
- **D8 — Занятия.** Учебные занятия с датой и типом.
- **D9 — Оценки.** Выставленные оценки студентов с привязкой к работе, преподавателю и дате.
- **D10 — Посещаемость.** Отметки о посещаемости с привязкой к занятию и студенту.
- **D11 — Типы работ.** Справочник типов работ.
- **D12 — Системы оценивания.** Справочник систем оценивания: пятибалльная, зачет/незачет, произвольная.
- **D13 — Типы занятий.** Справочник типов занятий.
- **D14 — Журнал аудита.** Записи обо всех изменениях данных в системе.

## 2.3 Информационные модели баз данных

Информационная модель базы данных спроектирована в методологии IDEF1X [3] и представлена на рисунке 2.3. Модель отражает логическую структуру данных, включающую семнадцать взаимосвязанных сущностей, соответствующих таблицам реляционной базы данных PostgreSQL.

*[ВСТАВИТЬ: Рисунок 2.3 — Информационная модель базы данных (IDEF1X, обновленная под 17 таблиц)]*

### 2.3.1 Сущность «Роли» (roles)

Сущность хранит справочник ролей пользователей системы и является независимой (корневой) сущностью. Каждая роль определяет набор прав доступа пользователя к функциям системы.

- **id** (PK) — целочисленный уникальный идентификатор роли, формируемый автоматически;
- **name** — строковое наименование роли, принимающее одно из четырех значений: 'admin', 'teacher', 'student', 'deanery'.

### 2.3.2 Сущность «Пользователи» (users)

Сущность хранит учетные данные всех пользователей системы и является центральной для обеспечения аутентификации и авторизации. Имеет идентифицирующую связь с сущностью «Роли».

- **id** (PK) — целочисленный уникальный идентификатор пользователя;
- **login** — строковый логин для входа в систему, уникальный в пределах таблицы;
- **password_hash** — строковый хэш пароля, сформированный алгоритмом bcrypt;
- **full_name** — строковое полное имя пользователя (фамилия, имя, отчество);
- **role_id** (FK) — внешний ключ, ссылающийся на сущность «Роли»;
- **created_at** — временная метка создания учетной записи, позволяющая отслеживать хронологию регистрации пользователей и использовать эту информацию в аудиторских целях;
- **updated_at** — временная метка последнего обновления учетной записи, фиксирующая любые изменения данных пользователя (смена пароля, блокировка, редактирование ФИО) для обеспечения прослеживаемости изменений;
- **is_blocked** — логический признак блокировки учетной записи, по умолчанию false.

### 2.3.3 Сущность «Группы» (groups)

Сущность представляет учебные группы студентов и является независимой.

- **id** (PK) — целочисленный уникальный идентификатор группы;
- **name** — строковое название группы, уникальное в пределах таблицы;
- **admission_year** — целочисленный год поступления группы, ограниченный диапазоном с 2000 по 2100 год.

### 2.3.4 Сущность «Студенты» (students)

Сущность представляет собой ассоциативную сущность, связывающую пользователя-студента с учебной группой. Имеет идентифицирующие связи с сущностями «Пользователи» и «Группы».

- **id** (PK) — целочисленный уникальный идентификатор записи;
- **user_id** (FK) — внешний ключ на сущность «Пользователи», каждому студенту соответствует ровно одна учетная запись;
- **group_id** (FK) — внешний ключ на сущность «Группы», указывает текущую группу студента.

### 2.3.5 Сущность «Преподаватели» (teachers)

Сущность представляет собой ассоциативную сущность, связывающую пользователя-преподавателя с кафедрой. Имеет идентифицирующую связь с сущностью «Пользователи».

- **id** (PK) — целочисленный уникальный идентификатор записи;
- **user_id** (FK) — внешний ключ на сущность «Пользователи»;
- **department** — строковое название кафедры, к которой относится преподаватель.

### 2.3.6 Сущность «Дисциплины» (disciplines)

Независимая сущность, представляющая учебные дисциплины.

- **id** (PK) — целочисленный уникальный идентификатор дисциплины;
- **name** — строковое название дисциплины.

### 2.3.7 Сущность «Назначения преподавателей» (teacher_assignments)

Ассоциативная сущность, реализующая связь «многие-ко-многим» между преподавателями, группами и дисциплинами с учетом семестра. При создании записи в данной сущности система автоматически порождает две записи в сущности «Журналы».

- **id** (PK) — целочисленный уникальный идентификатор назначения;
- **teacher_id** (FK) — внешний ключ на сущность «Преподаватели»;
- **group_id** (FK) — внешний ключ на сущность «Группы»;
- **discipline_id** (FK) — внешний ключ на сущность «Дисциплины»;
- **semester** — строковое обозначение семестра;
- **created_at** — временная метка создания назначения.

### 2.3.8 Сущность «Журналы» (journals)

Сущность представляет собой журнал успеваемости или журнал посещаемости, автоматически создаваемый при назначении преподавателя.

- **id** (PK) — целочисленный уникальный идентификатор журнала;
- **group_id** (FK) — внешний ключ на сущность «Группы»;
- **teacher_id** (FK) — внешний ключ на сущность «Преподаватели»;
- **discipline_id** (FK) — внешний ключ на сущность «Дисциплины»;
- **semester** — строковое обозначение семестра;
- **type** — строковый тип журнала: 'grades' или 'attendance';
- **created_at** — временная метка создания журнала.

### 2.3.9 Сущность «Типы работ» (work_types)

Независимый справочник типов учебных работ.

- **id** (PK) — целочисленный уникальный идентификатор;
- **name** — строковое наименование типа работы;
- **slug** — строковый символьный код типа работы.

### 2.3.10 Сущность «Системы оценивания» (grade_systems)

Независимый справочник систем оценивания.

- **id** (PK) — целочисленный уникальный идентификатор;
- **name** — строковое наименование системы;
- **description** — текстовое описание системы оценивания.

### 2.3.11 Сущность «Работы» (works)

Сущность представляет отдельную учебную работу в рамках журнала успеваемости.

- **id** (PK) — целочисленный уникальный идентификатор работы;
- **journal_id** (FK) — внешний ключ на сущность «Журналы»;
- **title** — строковое название работы;
- **work_type_id** (FK) — внешний ключ на сущность «Типы работ»;
- **grade_system_id** (FK) — внешний ключ на сущность «Системы оценивания»;
- **min_score** — числовое минимальное количество баллов;
- **max_score** — числовое максимальное количество баллов;
- **is_mandatory** — логический признак обязательности работы;
- **deadline** — дата срока сдачи работы;
- **display_order** — целочисленный порядок отображения;
- **is_active** — логический признак активности записи.

### 2.3.12 Сущность «Типы занятий» (lesson_types)

Независимый справочник типов учебных занятий.

- **id** (PK) — целочисленный уникальный идентификатор;
- **name** — строковое наименование типа занятия;
- **slug** — строковый символьный код.

### 2.3.13 Сущность «Занятия» (lessons)

Сущность представляет отдельное учебное занятие в рамках журнала.

Класс **Attendance** фиксирует статус посещаемости студента на конкретном занятии с возможными значениями: присутствует, отсутствует, уважительная причина.

Класс **AuditLog** обеспечивает ведение журнала изменений данных и содержит атрибуты: действие (создание, обновление, удаление), имя таблицы, предыдущее и новое состояние записи в формате JSONB, временную метку. Класс **Setting** представляет системные настройки в формате «ключ-значение».

### 2.4.1 Архитектура серверной части и использование классов

Серверная часть системы реализована на платформе Node.js версии 18 с использованием фреймворка Express.js. Архитектура сервера построена по модульному принципу с четким разделением ответственности между компонентами. Ключевой особенностью архитектуры является применение объектно-ориентированного подхода при реализации моделей предметной области: все модели спроектированы как ES6-классы с приватными полями, геттерами, сеттерами, статическими методами для работы с коллекциями и методами экземпляра для работы с конкретной записью.

Классы моделей инкапсулируют логику доступа к данным и бизнес-правила. Например, класс User содержит приватные поля `#id`, `#login`, `#passwordHash`, `#fullName` и открытые методы для получения и изменения данных. Статические методы классов, такие как `findById`, `findAll`, `create`, обеспечивают операции поиска и создания записей в базе данных. Методы экземпляра, такие как `save()`, `delete()`, `toJSON()`, позволяют сохранять, удалять и сериализовать объекты. Некоторые модели используют наследование: класс Student расширяет функциональность базового класса User, добавляя специфические атрибуты и методы, связанные с учебной группой.

Контроллеры реализованы в виде отдельных асинхронных функций, организованных по функциональным доменам. Сервисный слой в архитектуре отсутствует: контроллеры напрямую вызывают методы классов-моделей, что упрощает структуру проекта при текущем масштабе системы. Каждый функциональный домен имеет собственный файл маршрутизатора (auth.js, admin.js, grades.js, attendance.js и др.), связывающий HTTP-методы с соответствующими контроллерами.

Middleware системы включают:
- **authenticateToken** — извлекает JWT-токен из httpOnly-cookie, проверяет его валидность и подпись с помощью секретного ключа, помещает декодированные данные пользователя в объект запроса;
- **requireRole(...roles)** — проверяет соответствие роли пользователя из токена заданному списку разрешенных ролей, отклоняя запрос с кодом 403 при несоответствии;
- **checkLoginBlock** — отслеживает неудачные попытки входа с одного IP-адреса, блокируя доступ на 15 минут при превышении пяти попыток;
- **validateBody(schema)** — валидирует тело запроса по схеме Zod, гарантируя корректность входных данных;
- **errorHandler** — централизованно обрабатывает ошибки, формируя стандартизированный ответ и скрывая технические детали в production-режиме.

### 2.4.2 Графический клиент

Клиентская часть системы разработана с использованием библиотеки React версии 18 и языка TypeScript, что обеспечивает статическую типизацию компонентов и снижает количество ошибок на этапе разработки. Сборка проекта осуществляется с помощью инструмента Vite. Для стилизации интерфейса применяется фреймворк Tailwind CSS.

Клиентское приложение функционирует как одностраничное веб-приложение (SPA). Навигация между страницами осуществляется без перезагрузки браузера. Глобальное состояние аутентификации управляется через контекст AuthContext, доступный всем компонентам приложения. HTTP-запросы к серверу выполняются через библиотеку Axios с настроенным interceptor для обработки ошибок аутентификации.

Структура клиентской части организована по модульному принципу: директория pages/ содержит страницы приложения, components/ — переиспользуемые элементы интерфейса, context/ — глобальное состояние, services/ — модули взаимодействия с API, types/ — TypeScript-интерфейсы.

### 2.4.3 Настройка HTTPS и взаимодействие клиента с сервером

Взаимодействие между клиентской и серверной частями организовано с применением протокола HTTPS для клиентского dev-сервера и HTTP для серверного приложения. Данная конфигурация обусловлена особенностями разработки и не предполагает промышленного развертывания.

Клиентский dev-сервер Vite настроен на работу по протоколу HTTPS с использованием самоподписанных сертификатов localhost.pem и localhost-key.pem, расположенных в корневой директории проекта. Сервер прослушивает порт 5173. Все запросы к API, начинающиеся с префикса /api, проксируются на backend по HTTP на порт 3001. Таким образом, браузер взаимодействует с фронтендом по HTTPS, а фронтенд, в свою очередь, перенаправляет API-запросы на HTTP-бэкенд.

Сервер Express запускается на чистом HTTP (порт 3001) без использования TLS-сертификатов. Настройка CORS разрешает запросы как с http://localhost:5173, так и с https://localhost:5173. Обмен данными производится в формате JSON по архитектурному стилю REST.

Для всех запросов, требующих аутентификации, JWT-токен передается автоматически в httpOnly-cookie, что исключает возможность кражи токена через межсайтовый скриптинг. При успешной авторизации сервер устанавливает cookie с токеном, а клиент сохраняет информацию о пользователе в AuthContext и выполняет навигацию на Dashboard в соответствии с ролью.

## 3.1 Описание системы

Разработка информационной системы учета успеваемости студентов велась с применением современного стека веб-технологий, обеспечивающего высокую производительность, масштабируемость и безопасность приложения. Клиентская часть реализована на библиотеке React.js версии 18 с использованием языка TypeScript, обеспечивающего строгую типизацию и повышение надежности кода. Фреймворк Express.js предоставляет удобные механизмы маршрутизации, промежуточной обработки запросов и интеграции с различными модулями безопасности. React.js был выбран для создания динамичного и отзывчивого пользовательского интерфейса с компонентным подходом к разработке, что упрощает поддержку и расширение функциональности. PostgreSQL обеспечивает надежное хранение данных, полную поддержку транзакций, расширенные возможности работы с JSONB-структурами и соответствие стандартам SQL.

Архитектура системы построена по классической трехуровневой схеме клиент-сервер с четким разделением на уровень представления (frontend), уровень бизнес-логики (backend) и уровень хранения данных (база данных PostgreSQL). База данных обеспечивает следующие механизмы целостности и автоматизации:

## 3.3 Серверная часть

Серверная часть приложения организована в директории server/src/ по модульному принципу. Каждый функциональный домен выделен в отдельный файл маршрутизатора и соответствующие модули контроллеров и моделей.

*[ВСТАВИТЬ: Рисунок 3.3 — Структура директории server/src/ (скриншот проводника или VS Code)]*

Реализованы следующие маршрутизаторы и их конечные точки:

**auth.js** — базовый путь /api/auth:
- POST /login — аутентификация пользователя с проверкой логина и пароля, формированием JWT-токена и установкой cookie. Перед обработкой запроса выполняется проверка rate limiting через middleware checkLoginBlock.
- POST /logout — завершение сессии с удалением cookie и очисткой токена.
- GET /me — получение данных текущего авторизованного пользователя на основе валидности JWT-токена.

**admin.js** — базовый путь /api/admin, все маршруты защищены middleware authenticateToken и requireRole('admin'):
- GET /users — получение списка всех пользователей;
- POST /users — создание пользователя с валидацией тела запроса через createUserSchema;
- PUT /users/:id — обновление данных пользователя с валидацией через updateUserSchema;
- DELETE /users/:id — удаление пользователя;
- PUT /users/:id/group — изменение группы студента;
- PUT /users/:id/department — изменение кафедры преподавателя;
- GET /audit-logs — получение журнала аудита;
- GET /groups, POST /groups, PUT /groups/:id, DELETE /groups/:id — CRUD-операции с учебными группами.

**assignments.js** — базовый путь /api/assignments:
- GET / — получение списка назначений преподавателей;
- POST / — создание назначения (только admin);
- DELETE /:id — удаление назначения (только admin).

**attendance.js** — базовый путь /api/attendance:
- GET / — получение данных посещаемости;
- POST / — установка статуса посещаемости (teacher, admin, deanery);
- GET /stats/:journalId — статистика посещаемости по журналу.

**dean.js** — базовый путь /api/dean (admin, deanery):
- GET /summary/groups — сводка по группам;
- GET /summary/disciplines — сводка по дисциплинам;
- GET /summary/students/:groupId — сводка по студентам группы.

**disciplines.js** — базовый путь /api/disciplines:
- GET / — список дисциплин;
- POST / — создание дисциплины (admin);
- DELETE /:id — удаление дисциплины (admin).

**grades.js** — базовый путь /api/grades:
- GET / — получение оценок;
- POST / — выставление оценки (teacher, admin, deanery);

Система аутентификации реализована на основе JWT-токенов. При успешной авторизации сервер формирует токен с использованием библиотеки jsonwebtoken, в payload которого записываются идентификатор пользователя, его логин и роль. Токен подписывается секретным ключом и устанавливается в httpOnly-cookie с флагом secure и ограниченным сроком действия 24 часа. При каждом последующем запросе middleware authenticateToken извлекает токен из cookie, проверяет его валидность и подпись, после чего помещает декодированные данные пользователя в объект запроса req.user. Middleware requireRole сравнивает роль пользователя с требуемым списком ролей и возвращает ошибку 403 при несоответствии.

### 3.3.1 Резервное копирование и восстановление

Механизм резервного копирования реализован внешними PowerShell-скриптами и не входит в состав серверного приложения Express, однако является неотъемлемой частью инфраструктуры системы. Скрипт backup.ps1 обеспечивает создание полных резервных копий базы данных PostgreSQL в формате plain-text SQL-дампов. При запуске скрипт последовательно выполняет следующие операции: определяет путь к утилите pg_dump через команду Get-Command или перебор стандартных директорий установки PostgreSQL (C:\Program Files\PostgreSQL\{18..15}\bin); считывает параметры подключения из файла server/.env (хост, порт, имя базы, пользователь, пароль); устанавливает переменную окружения PGPASSWORD для неинтерактивной аутентификации; формирует имя выходного файла с временной меткой в формате backup_electronic_journal_YYYYMMDD_HHMMSS.sql; выполняет pg_dump с ключами -h, -p, -U, -F p, -f; проверяет код возврата и выводит результат пользователю.

Скрипт restore.ps1 обеспечивает восстановление базы данных из резервной копии с механизмом защиты от потери текущих данных. При запуске скрипт выполняет следующие операции: считывает параметры подключения из server/.env; определяет пути к утилитам psql, pg_dump, dropdb, createdb; выводит нумерованный список файлов .sql из директории backups/; предлагает пользователю выбрать резервную копию по номеру; создает предварительную резервную копию текущего состояния базы данных перед восстановлением (для возможности отката); принудительно удаляет существующую базу данных с помощью dropdb --force (ключ --force завершает все активные соединения); создает новую пустую базу данных через createdb; восстанавливает данные из выбранного файла с помощью psql -f. Использование dropdb --force вместо команды DROP DATABASE внутри psql обусловлено тем, что операция DROP DATABASE не может выполняться внутри транзакции, в то время как dropdb является внешней утилитой, не ограниченной этим правилом.

# 4 ТЕСТИРОВАНИЕ

- создавать учебные работы, нажав кнопку «Добавить работу» и указав название, тип, систему оценивания и максимальный балл;
- выставлять оценки студентам в соответствующих ячейках таблицы;
- просматривать автоматически рассчитанный средний балл и итоговую оценку каждого студента.

*[ВСТАВИТЬ: Рисунок А.3 — Журнал успеваемости (скриншот)]*

*[ВСТАВИТЬ: Рисунок А.4 — Журнал посещаемости (скриншот)]*

# Руководство администратора

## Б.1 Управление пользователями

Пользователи с ролью «Администратор» имеют доступ к панели администратора по адресу /admin. В разделе «Пользователи» отображается таблица всех учетных записей системы. Администратор может:
- создавать новых пользователей, указывая логин, ФИО, пароль, роль, а для студента — группу, для преподавателя — кафедру;
- редактировать данные существующих пользователей;
- удалять пользователей из системы.

*[ВСТАВИТЬ: Рисунок Б.1 — Панель управления пользователями (скриншот)]*

## Б.2 Управление справочниками

Раздел «Группы» позволяет управлять учебными группами: добавлять новые группы с указанием названия и года поступления, редактировать существующие, удалять.

Раздел «Дисциплины» позволяет управлять справочником учебных дисциплин.

*[ВСТАВИТЬ: Рисунок Б.2 — Управление группами (скриншот)]*

## Б.3 Назначение преподавателей

В разделе «Назначения» (/assignments) администратор создает связь между преподавателем, группой, дисциплиной и семестром. При создании назначения система автоматически формирует два журнала: успеваемости и посещаемости. Удаление назначения приводит к удалению связанных журналов.

*[ВСТАВИТЬ: Рисунок Б.3 — Страница назначений (скриншот)]*

## Б.4 Резервное копирование

Для создания резервной копии базы данных необходимо запустить скрипт scripts/backup.ps1 в PowerShell. Скрипт автоматически обнаруживает утилиту pg_dump, читает учетные данные из файла server/.env и создает SQL-дамп в директории backups/.

*[ВСТАВИТЬ: Рисунок Б.4 — Запуск скрипта backup.ps1 (скриншот)]*

Для восстановления базы данных необходимо запустить скрипт scripts/restore.ps1. Скрипт выводит список доступных резервных копий, предлагает выбрать нужную, создает предварительную копию текущего состояния, удаляет и пересоздает базу данных с помощью утилит dropdb и createdb, после чего восстанавливает данные из выбранного файла с помощью psql.

*[ВСТАВИТЬ: Рисунок Б.5 — Запуск скрипта restore.ps1 (скриншот)]*

## Б.5 Системные требования

**Минимальные аппаратные требования:**
- процессор: Intel Core i3 / AMD Ryzen 3 (2 ядра и более);
- оперативная память: 4 ГБ;
- накопитель: 5 ГБ свободного места;
- операционная система: Windows 10/11.

**Программное обеспечение:**
- Node.js 18+ — для запуска серверной и клиентской частей;
- PostgreSQL 15+ — локальная база данных;
- npm 9+ — менеджер пакетов.

## Б.6 Локальный запуск системы

1. Установить PostgreSQL и создать базу данных electronic_journal.
2. Создать таблицы с помощью SQL-скрипта инициализации.
3. Настроить подключение в файле server/.env (DB_HOST, DB_PORT, DB_NAME, DB_USER, DB_PASSWORD, JWT_SECRET).
4. Установить зависимости сервера: cd server && npm install.
5. Установить зависимости клиента: cd client && npm install.
6. Запустить сервер: cd server && npm run dev.
7. Запустить клиент: cd client && npm run dev.
8. Открыть в браузере: https://localhost:5173.
