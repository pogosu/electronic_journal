# ВВЕДЕНИЕ

Целью курсовой работы является создание системы, представленной в виде клиент-серверного приложения для учета успеваемости и посещаемости студентов. В процессе выполнения курсовой работы были выполнены следующие этапы:

- разработка технического задания;
- оценка рисков системы;
- разработка структурной, функциональной и информационной моделей;
- разработка программы;
- тестирование и доработка программы.

Курсовая работа направлена на закрепление теоретических знаний, полученных в ходе изучения дисциплин, связанных с программированием, проектированием программных систем и разработкой пользовательского интерфейса, а также на развитие практических навыков работы с инструментами разработки.

Разрабатываемая информационная система (ИС) представляет собой электронный журнал успеваемости, предназначенный для автоматизации процессов учета оценок и посещаемости студентов в высшем учебном заведении. Система обеспечивает взаимодействие между преподавателями, студентами и административным персоналом, а также поддерживает ведение журналов по дисциплинам, формирование отчетов и резервное копирование данных.

Основной целью системы является предоставление преподавателям удобного инструмента для ведения журналов успеваемости, студентам — возможности контроля своей учебной деятельности, а администраторам — средств управления учебным процессом.

В данной предметной области выделяются следующие группы сущностей, каждая из которых представляет собой базу данных для хранения информации:

- первая группа: пользователи, группы студентов, кафедры, преподаватели, студенты;
- вторая группа: дисциплины, расписание занятий, оценки, посещаемость.

В разрабатываемой ИС реализуются следующие механизмы автоматизации:

- ведение электронных журналов — преподаватель вносит оценки и отметки о посещаемости в электронную форму, система автоматически сохраняет данные и пересчитывает итоговые показатели;
- фильтрация и поиск — предоставление возможности фильтрации журналов по группе, дисциплине, периоду и преподавателю, а также поиска студентов по фамилии или номеру группы;
- разграничение прав доступа — система обеспечивает доступ к функциям в соответствии с ролью пользователя (администратор, преподаватель, студент, деканат);
- резервное копирование — автоматизированное создание резервных копий базы данных с возможностью восстановления.

В системе выделяются следующие функциональные возможности преподавателя:

- просмотр списка закрепленных за ним журналов;
- ведение журнала успеваемости (выставление оценок за занятия, промежуточная и итоговая аттестации);
- ведение журнала посещаемости (отметка присутствующих и отсутствующих);
- просмотр статистики по группе и дисциплине.

В системе выделяются следующие функциональные возможности студента:

- просмотр собственной успеваемости и посещаемости по дисциплинам;
- просмотр итоговых оценок и среднего балла.

В системе выделяются следующие функциональные возможности администратора:

- управление учетными записями пользователей (создание, редактирование, удаление);
- управление справочниками (группы, кафедры, дисциплины);
- назначение преподавателей на дисциплины и группы;
- просмотр всех журналов системы.

В рамках данной ИС не реализуются следующие возможности:

- интеграция с внешними системами (1С, LMS Moodle и т.д.);
- автоматическая рассылка уведомлений студентам и преподавателям по электронной почте;
- мобильное приложение;
- экспорт данных в формате PDF/Excel;
- автоматическое формирование приказов о переводе/отчислении.

Основной функцией данного процесса является учет успеваемости и посещаемости студентов в электронном журнале. Процесс начинается с введенного URL-адреса сайта, с помощью которого пользователь попадает в информационную систему. Авторизованный пользователь (преподаватель, студент или администратор) взаимодействует с системой в соответствии со своей ролью.

Для корректной работы системы необходимы определенные входные данные и условия. В первую очередь, должны быть обеспечены доступность базы данных PostgreSQL и сервера приложений Node.js, которые являются фундаментом работоспособности системы. Также необходимо соблюдение ФЗ-152 «О персональных данных» при поступлении данных пользователей в момент их регистрации в системе. Важным элементом поддержки является руководство пользователя, которое помогает корректно взаимодействовать с программой. Система должна обладать интуитивным интерфейсом, позволяющим пользователю за наименьшее время разобраться в предоставляемых ему функциональных возможностях.

В результате выполнения функции формируются новые данные, которые вносятся в базу данных: оценки студентов, отметки о посещаемости. После завершения работы пользователь может выйти из своей учетной записи.

Таким образом, процесс «Учет успеваемости и посещаемости студентов» представляет собой комплексную функцию, которая обеспечивает переход пользователя в информационную систему, ведение журналов, выставление оценок и отметок о посещаемости при условии соблюдения доступности сервера и баз данных.

Для более детального представления работы системы была выполнена декомпозиция черного ящика. В результате основная функция разбита на последовательность подфункций, каждая из которых отвечает за конкретный этап работы с электронным журналом. Декомпозиция представлена в приложении Г.

Основные подфункции:

- **Авторизоваться в системе (А1).** Пользователь вводит логин и пароль на странице авторизации. Система проверяет учетные данные в базе данных, формирует JWT-токен и устанавливает сессию. Выходом являются активная сессия и перенаправление в личный кабинет.
- **Управление учетными записями (А2).** Администратор создает, редактирует и удаляет учетные записи пользователей. Включает подфункции: создание пользователя (А2.1), редактирование данных пользователя (А2.2), удаление пользователя (А2.3).
- **Управление справочниками (А3).** Администратор управляет группами, кафедрами, дисциплинами и расписанием. Включает подфункции: управление группами (А3.1), управление дисциплинами (А3.2), назначение преподавателей (А3.3).
- **Ведение журнала успеваемости (А4).** Преподаватель открывает закрепленный за ним журнал, выставляет оценки за занятия, промежуточную и итоговую аттестацию. Система автоматически пересчитывает средний балл и итоговую оценку.
- **Ведение журнала посещаемости (А5).** Преподаватель отмечает присутствующих и отсутствующих студентов на занятии. Система подсчитывает процент посещаемости.
- **Просмотр успеваемости (А6).** Студент просматривает свои оценки и посещаемость по всем дисциплинам. Преподаватель и администратор могут просматривать журналы с фильтрацией.
- **Формирование отчетов (А7).** Система формирует статистику успеваемости и посещаемости по группе, дисциплине или периоду.

Основные элементы модели:

- **Неавторизованный пользователь.** Пользователь, который еще не выполнил вход в систему. Он может просматривать страницу авторизации. Для доступа к журналам и отчетам необходима авторизация.
- **Преподаватель.** Участник системы, который ведет журналы успеваемости и посещаемости по закрепленным за ним дисциплинам и группам: выставляет оценки, отмечает посещаемость, просматривает статистику.
- **Студент.** Участник системы, который просматривает свою успеваемость и посещаемость по дисциплинам. Имеет доступ только к собственным данным.

- **Деканат.** Участник системы с правами просмотра журналов и формирования отчетов без возможности редактирования оценок.

Основные процессы:

- **Процесс 1 — Авторизация.** Проверка учетных данных, формирование JWT-токена, установка cookie.
- **Процесс 2 — Управление пользователями.** CRUD-операции с учетными записями (только администратор).
- **Процесс 3 — Управление справочниками.** CRUD-операции с группами, кафедрами, дисциплинами, расписанием.
- **Процесс 4 — Ведение журнала успеваемости.** Выставление оценок, автоматический пересчет среднего балла и итоговой оценки.
- **Процесс 5 — Ведение журнала посещаемости.** Отметка присутствия/отсутствия, подсчет процента посещаемости.
- **Процесс 6 — Просмотр журналов.** Фильтрация по группе, дисциплине, периоду, преподавателю; поиск студента по ФИО.
- **Процесс 7 — Формирование отчетов.** Статистика успеваемости и посещаемости.
- **Процесс 8 — Резервное копирование.** Создание и восстановление резервных копий БД.

Для системы спроектирована диаграмма компонентов, представленная на рисунке 2.6. На данной диаграмме показана общая структура приложения [4].

Основу серверной части системы составляет набор компонентов, обеспечивающих бизнес-логику электронного журнала. Сервер включает в себя компоненты: Auth, User, Group, Subject, Teacher, Student, Grade, Attendance, Schedule, Backup. Каждый из этих компонентов отвечает за свою функциональную область и взаимодействует с соответствующими таблицами базы данных и клиентскими компонентами.

Компонент **Auth** управляет процессами аутентификации и авторизации. Он обеспечивает проверку учетных данных, формирование JWT-токенов, контроль срока действия сессии и выход из системы. Через этот компонент клиентские страницы получают подтверждение авторизации и права доступа.

Компонент **Grade** отвечает за обработку оценок. Он обеспечивает создание, редактирование и удаление оценок, автоматический пересчет среднего балла и итоговой оценки по дисциплине. Компонент Grade взаимодействует с таблицей Grades базы данных PostgreSQL.

Компонент **Attendance** управляет учетом посещаемости. Он обрабатывает отметки о присутствии и отсутствии студентов, подсчитывает процент посещаемости и формирует статистику.

Компоненты **User**, **Group**, **Subject** и **Teacher** взаимодействуют с соответствующими таблицами базы данных. Эти компоненты обеспечивают CRUD-операции для управления учетными записями, группами, дисциплинами и назначениями преподавателей.

Компонент **Backup** обеспечивает создание резервных копий базы данных через вызов PowerShell-скрипта pg_dump. Он взаимодействует с операционной системой сервера и файловой системой для сохранения файлов резервных копий.

База данных PostgreSQL является внешним хранилищем для всех серверных компонентов. Она обеспечивает сохранность всех данных системы и предоставляет данные для отображения в журналах.

На стороне клиента (графический интерфейс) систему составляют компоненты страниц: LoginPage, DashboardPage, JournalPage, AdminPage, ProfilePage. Каждый из этих компонентов имеет соответствующий пользовательский интерфейс. Через эти интерфейсы пользователь получает доступ ко всем функциям приложения.

Компонент **JournalPage** отображает журнал успеваемости и посещаемости. Через интерфейс преподаватель может выставлять оценки и отмечать посещаемость, а студент — просматривать свои данные.

Компонент **AdminPage** обеспечивает работу панели администратора: управление пользователями, группами, дисциплинами и просмотр всех журналов.

Компонент **DashboardPage** отображает главную страницу с перечнем доступных журналов, фильтрами и поиском.

### 2.4.1 Серверная часть

Серверная часть написана на JavaScript (Node.js 18+) с использованием фреймворка Express.js. Основные компоненты:

- **Маршрутизаторы (Routes/Controllers)** — обрабатывают HTTP-запросы от клиента и передают их в сервисы.
- **Сервисы (Services)** — содержат бизнес-логику системы (обработка оценок, пересчет среднего балла, формирование отчетов).
- **Модели (Models/Prisma)** — обеспечивают доступ к базе данных через ORM Prisma.
- **Middleware** — промежуточные обработчики: аутентификация (authMiddleware), проверка ролей (roleMiddleware), ограничение частоты запросов (rateLimit).
- **Валидаторы (Validators/Zod)** — схемы валидации входных данных.

### 2.4.2 Графический клиент

Клиентская часть разработана на React 18 с использованием TypeScript и следующих библиотек:

- **Vite** — инструмент сборки проекта;
- **React Router** — для маршрутизации и навигации;
- **Tailwind CSS** — для стилизации интерфейса;
- **Axios** — для HTTP-запросов к серверу;
- **Zustand / React Context** — для управления состоянием авторизации.

Основные клиентские компоненты находятся в директории client/src/:
- pages/ — страницы приложения (LoginPage, DashboardPage, JournalPage, AdminPage);
- components/ — переиспользуемые компоненты интерфейса (таблицы журналов, формы, модальные окна);
- hooks/ — пользовательские React-хуки;
- services/ — модули для взаимодействия с REST API.

### 2.4.3 Взаимодействие между серверными и клиентскими компонентами

Компонент Auth на сервере взаимодействует с клиентским компонентом LoginPage. Auth проверяет учетные данные, формирует JWT-токен и передает его в cookie. LoginPage перенаправляет пользователя в DashboardPage.

Компонент Grade на сервере взаимодействует с клиентским компонентом JournalPage. Grade обрабатывает запросы на создание и обновление оценок, а JournalPage отображает текущие оценки и позволяет их редактировать.

Компоненты User, Group, Subject на сервере взаимодействуют с клиентским компонентом AdminPage. Они предоставляют данные для управления справочниками.

# 3 РЕАЛИЗАЦИЯ

## 3.1 Описание системы

Для разработки информационной системы учета успеваемости была выбрана трехуровневая архитектура клиент-серверного приложения. В качестве серверной платформы используется Node.js с фреймворком Express.js. Клиентская часть реализована на библиотеке React.js с использованием языка TypeScript. В качестве системы управления базами данных выбрана PostgreSQL.

Выбор Node.js и Express.js обусловлен их высокой производительностью при обработке асинхронных запросов, простотой развертывания и удобной работой с JSON. React.js был выбран для создания динамичного и отзывчивого пользовательского интерфейса с минимальным временем перерисовки страниц. TypeScript обеспечивает статическую типизацию, что снижает количество ошибок на этапе разработки. PostgreSQL обеспечивает надежное хранение данных, поддержку сложных запросов и соответствие стандартам SQL.

Система построена по принципу клиент-сервер с четким разделением на уровень представления (frontend), бизнес-логики (backend) и хранения данных (база данных).

# ПРИЛОЖЕНИЕ А (обязательное)
# Руководство пользователя

## А.1 Вход в систему

После перехода на сайт информационной системы пользователю открывается страница авторизации. Для работы с журналами необходимо авторизоваться. Для этого нужно ввести логин и пароль, затем нажать кнопку «Войти».

*[ВСТАВИТЬ: Рисунок А.1 — Форма авторизации (скриншот)]*

При успешной авторизации в верхнем меню отобразится ФИО пользователя, а также станут доступны разделы в соответствии с ролью.

## А.2 Работа с журналами (для преподавателя)

После авторизации преподаватель попадает на страницу Dashboard, где отображается список закрепленных за ним журналов. Для открытия журнала необходимо кликнуть по соответствующей карточке.

*[ВСТАВИТЬ: Рисунок А.2 — Список журналов (скриншот)]*

В журнале отображается таблица со списком студентов группы. Преподаватель может:
- вводить оценки за занятия в соответствующие ячейки;
- выставлять оценки промежуточной и итоговой аттестации;
- просматривать автоматически пересчитанный средний балл и итоговую оценку.

*[ВСТАВИТЬ: Рисунок А.3 — Журнал успеваемости (скриншот)]*

Переключение на вкладку «Посещаемость» позволяет отмечать присутствие или отсутствие студентов.

*[ВСТАВИТЬ: Рисунок А.4 — Журнал посещаемости (скриншот)]*

## А.3 Просмотр успеваемости (для студента)

Студент после авторизации видит только свои собственные оценки и посещаемость по всем дисциплинам.

*[ВСТАВИТЬ: Рисунок А.5 — Просмотр успеваемости студентом (скриншот)]*

## А.4 Фильтрация и поиск

На странице Dashboard доступны фильтры:
- по группе;
- по дисциплине;
- по периоду (семестру);
- по преподавателю.

Также реализован поиск студента по фамилии или номеру группы.

*[ВСТАВИТЬ: Рисунок А.6 — Фильтры и поиск (скриншот)]*

## А.5 Выход из системы

Для выхода из системы необходимо нажать кнопку «Выйти» в верхнем меню. После этого сессия завершается и происходит перенаправление на страницу авторизации.

# ПРИЛОЖЕНИЕ Б (обязательное)
# Руководство администратора

## Б.1 Управление пользователями

Пользователи с ролью «Администратор» имеют доступ к панели администратора. Переход осуществляется через пункт «Администрирование» в верхнем меню.

В разделе «Пользователи» администратор может:
- просматривать список всех зарегистрированных пользователей;
- создавать новых пользователей (указать логин, ФИО, пароль, роль, группу/кафедру);
- редактировать данные существующих пользователей;
- удалять пользователей.

*[ВСТАВИТЬ: Рисунок Б.1 — Панель управления пользователями (скриншот)]*

## Б.2 Управление справочниками

Раздел «Группы» позволяет управлять студенческими группами: добавлять, редактировать название и кафедру, удалять.

Раздел «Дисциплины» позволяет управлять перечнем учебных дисциплин.

*[ВСТАВИТЬ: Рисунок Б.2 — Управление группами (скриншот)]*

## Б.3 Назначение преподавателей

В разделе «Назначения» администратор закрепляет преподавателя за группой и дисциплиной. Только после назначения преподаватель видит соответствующий журнал в своем списке.

*[ВСТАВИТЬ: Рисунок Б.3 — Назначение преподавателя (скриншот)]*

## Б.4 Резервное копирование

Для создания резервной копии базы данных необходимо запустить скрипт backup.ps1, расположенный в директории scripts/. Скрипт создает SQL-дамп в директории backups/.

*[ВСТАВИТЬ: Рисунок Б.4 — Запуск скрипта резервного копирования (скриншот)]*

Для восстановления базы данных из резервной копии необходимо запустить скрипт restore.ps1. Скрипт выведет список доступных резервных копий, создаст предварительную копию текущего состояния, удалит и пересоздаст базу данных, а затем восстановит данные из выбранной копии.

*[ВСТАВИТЬ: Рисунок Б.5 — Запуск скрипта восстановления (скриншот)]*

## Б.5 Системные требования

Минимальные системные требования:
- процессор: Intel Core i3 / AMD Ryzen 3 (2 ядра);
- оперативная память: 4 ГБ;
- накопитель: 5 ГБ свободного места;
- операционная система: Windows 10/11.

Требования по программному обеспечению:
- Node.js 18+ — для запуска серверной и клиентской частей;
- PostgreSQL 15+ — локальная база данных;
- pgAdmin (опционально) — графическое управление БД;
- Visual Studio Code — для разработки и запуска.

## Б.6 Локальный запуск

1. Установить PostgreSQL и создать базу данных electronic_journal.
2. Настроить подключение в файле server/.env (DATABASE_URL).
3. Выполнить миграции Prisma: npx prisma migrate dev.
4. Запустить сервер: cd server && npm run dev.
5. Запустить клиент: cd client && npm run dev.
6. Открыть в браузере: https://localhost:5173.
